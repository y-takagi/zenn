## はじめに

以前の記事で「フォーム上では数値も文字列で扱い、Zodの`transform`でバリデーション時に変換する」アプローチを紹介しました。

https://zenn.dev/genda_jp/articles/7eb105e892f458

このアプローチは確かに機能しますが、バリデーション前の状態（`field.state.value`）は`string`のまま扱われるため、リアルタイムに数値として利用したい場面で型変換が必要になります。

この記事では別のアプローチとして、**変換処理をInputコンポーネント側に持たせる**方法を紹介します。

## 前記事アプローチの課題

前記事のアプローチをおさらいします。

```ts:user-form.ts
export const formSchema = z.object({
  name: z.string().min(1),
  age: z.string().min(1).transform(Number).pipe(z.number()),
})

export type FormInputSchema = z.input<typeof formSchema>;
// { name: string, age: string }  ← ageはstring

export type FormOutputSchema = z.output<typeof formSchema>;
// { name: string, age: number }  ← バリデーション後はnumber
```

`onSubmit`で受け取る`value`は`FormOutputSchema`なので`age`は`number`型ですが、フォームの状態（`field.state.value`）は`FormInputSchema`由来のため`string`です。

リアルタイムに合計値を計算するような場面では、明示的な変換が必要になります。

```tsx
// form.Subscribeで値を購読しても、ageはstringのまま
<form.Subscribe selector={(state) => state.values.age}>
  {(age) => {
    // ageはstring型なので、numberとして使うにはNumber()が必要
    const ageAsNumber = Number(age);
    return <p>年齢: {ageAsNumber}</p>;
  }}
</form.Subscribe>
```

## 解決策: InputコンポーネントにNumber変換を持たせる

`value`と`onChange`のインターフェースを`number | null`にしたNumberInputコンポーネントを作り、**内部で`string`↔`number`の変換を行う**アプローチです。

```tsx:NumberInput.tsx
interface NumberInputProps extends Omit<React.InputHTMLAttributes<HTMLInputElement>, 'value' | 'onChange' | 'type'> {
  value?: number | null | undefined;
  onChange?: (value: number | null) => void;
}

export function NumberInput({ value, onChange, ...rest }: NumberInputProps) {
  const handleChange = (e: React.ChangeEvent<HTMLInputElement>) => {
    const raw = e.target.value;
    onChange?.(raw.length > 0 ? Number(raw) : null);
  };

  return (
    <input
      type="number"
      value={value == null ? '' : String(value)}
      onChange={handleChange}
      {...rest}
    />
  );
}
```

ポイントは2つです：

- **表示時**: `number | null` → `string`（`null`/`undefined`は空文字に変換）
- **変更時**: `string` → `number | null`（空文字は`null`に変換）

HTMLの`<input>`要素が`string`しか扱えないという制約をコンポーネントの内部に閉じ込め、外側にはアプリケーションのドメインに合った型（`number | null`）を公開します。

## フォームスキーマがシンプルになる

このアプローチでは、フォームの状態が最初から`number | null`なので、Zodスキーマに`transform`が不要です。

```ts:user-form.ts
export const formSchema = z.object({
  name: z.string().min(1),
  age: z.number().nullable(),  // transformなし！
})

export type FormInputSchema = z.input<typeof formSchema>;
// { name: string, age: number | null }

export type FormOutputSchema = z.output<typeof formSchema>;
// { name: string, age: number | null }
// inputとoutputの型が同じ！
```

`z.input`と`z.output`が同じ型になるため、バリデーション前後で型の扱いを意識する必要がなくなります。

## フォーム全体の実装

```ts:user-form.ts
const formSchema = z.object({
  name: z.string().min(1),
  age: z.number({ required_error: '年齢を入力してください' }).int().nonnegative(),
})

type FormValues = z.input<typeof formSchema>;

export function createDefaultFormValue(user: User | undefined): FormValues {
  return {
    name: user?.name ?? '',
    age: user?.age ?? null,  // null で初期化できる
  }
}
```

```tsx:UserForm.tsx
function UserForm({ user }: { user: User | undefined }) {
  const form = useForm({
    defaultValues: createDefaultFormValue(user),
    validators: { onChange: formSchema },
    onSubmit: ({ value }) => {
      // value.age は number 型（変換不要）
      console.log(value.age);
    },
  });

  return (
    <form onSubmit={(e) => { e.preventDefault(); form.handleSubmit() }}>
      <form.Field name="name">
        {(field) => (
          <input
            value={field.state.value}
            onChange={(e) => field.handleChange(e.target.value)}
          />
        )}
      </form.Field>
      <form.Field name="age">
        {(field) => (
          // NumberInput が number | null を受け取り、number | null を返す
          <NumberInput
            value={field.state.value}
            onChange={(v) => field.handleChange(v)}
          />
        )}
      </form.Field>
    </form>
  )
}
```

## リアルタイムに数値として扱える

このアプローチの恩恵が特に出るのは、フォームの値をリアルタイムに使って何か計算する場面です。

```tsx
// form.Subscribeで購読した値がそのままnumber | null
<form.Subscribe selector={(state) => state.values}>
  {(values) => {
    // 型変換なしでそのまま計算できる
    const total = (values.price ?? 0) * (values.quantity ?? 0);
    return <p>合計金額: {total.toLocaleString()}円</p>;
  }}
</form.Subscribe>
```

前記事のアプローチでは`Number(values.price)`のような変換が必要でしたが、ここでは不要です。

## 2つのアプローチの比較

|                          | transformアプローチ       | Inputコンポーネントアプローチ |
| ------------------------ | ------------------------- | ----------------------------- |
| スキーマ定義             | `transform` が必要        | `z.number()` のみ             |
| `field.state.value` の型 | `string`                  | `number \| null`              |
| リアルタイム計算         | `Number()` での変換が必要 | そのまま使える                |
| 変換の責任               | Zodスキーマ               | Inputコンポーネント           |
| 再利用性                 | スキーマごとに定義        | コンポーネントを使い回せる    |

どちらのアプローチも正しく機能しますが、数値フィールドが複数あったり、リアルタイムに値を使う場面が多いプロジェクトでは、Inputコンポーネントに変換を持たせるアプローチのほうがシンプルで扱いやすいと感じています。

## まとめ

Reactの数値入力フォームをシンプルに実装する方法として、NumberInputコンポーネントに`string`↔`number`の変換を持たせるアプローチを紹介しました。

- **InputコンポーネントのI/Fを`number | null`にする**: `<input>`の`string`制約をコンポーネント内部に隠蔽する
- **Zodスキーマがシンプルになる**: `transform`が不要で、`z.input`と`z.output`の型が一致する
- **常に期待する型で値を扱える**: バリデーション前後・リアルタイム問わず`number | null`として扱える

TanStack Formに限らず、React Hook FormなどのフォームライブラリやZodと組み合わせても同様に使えます。ぜひ参考にしてください。
