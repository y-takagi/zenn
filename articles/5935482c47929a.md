## はじめに

フロントエンドとバックエンドを別々の Git リポジトリで管理しているチームは多いです。
リポジトリを分けることで独立したデプロイ・CI/CD・レビューが実現できる一方、
AI エージェントを使った開発では大きな壁にぶつかることがあります。

本記事では、社内プロダクト（バックエンド `myapp-be` / フロントエンド `myapp-fe`）の開発で
**既存の CI/CD・PR・ブランチ運用を一切変えずに**、AI エージェントが両リポを横断して開発できる環境を
git submodule superproject を使って構築した記録を紹介します。

## 課題：複数リポ構成で AI agent が使いにくい

Claude Code や Cursor などの AI エージェントは、**デフォルトでは開いているワークスペース外のコードを参照できません**。

たとえばバックエンドの API レスポンス構造を変更する際、フロントエンド側での影響確認や実装も同時に行いたいケースがあります。
しかし `myapp-be` のみを開いた状態でエージェントに依頼すると、`myapp-fe` のコードが見えていないまま答えが返ってきます。
逆に `myapp-fe` を開くと、今度は `be` 側の API 実装が見えません。

かといって**完全なモノレポへの移行は大きなコスト**を伴います。
ビルドツールの統一、CI/CD パイプラインの再設計、PR 運用の変更、既存の自動デプロイとの整合性確認など、すでに動いている仕組みを全て作り直すリスクを取るのは難しい状況でした。

**求めていたのは「既存の各リポを完全にそのまま維持しながら、AI エージェントには両方を同時に見せる」方法です。**

## 解法の選択：なぜ submodule superproject か

### モノレポツールは既存構成への変更が大きい

Turborepo や Nx は強力なモノレポツールですが、導入には各リポへの設定ファイル追加・ビルドツール統一が必要になります。
既存の独立した CI/CD や PR 運用と共存させるのは難しく、「既存リポには手を触れない」という制約を満たせません。

### 親ディレクトリアプローチは git worktree と相性が悪い

最もシンプルに思えるのは「両リポを含む親ディレクトリを作り、そこから Claude Code を起動する」方法です。

```
~/projects/myapp-workspace/
├── myapp-be/
└── myapp-fe/
```

設定ゼロで AI エージェントが両リポを参照できますが、2つの問題があります。

**1. git worktree との対応が取れない**

フィーチャーブランチで作業するとき、`myapp-be` と `myapp-fe` それぞれで worktree を別々に切り出す必要があります。
be の `feat/xxx` worktree と fe の `feat/xxx` worktree を手動で対応付けて管理しなければならず、
機能開発のたびに「どの worktree 同士が対になっているか」を意識する手間が生まれます。

**2. 親ディレクトリは git リポジトリではない**

親ディレクトリ自体は git で管理されていないため、Claude Code の `/add-dir` で追加した状態はセッション固有になります。
セッションをまたぐたびに再設定が必要で、チームで共有できる構成として定着しにくいです。

git submodule superproject はこれらをどちらも解決します。
superproject の worktree を1つ切り出せば、`be/` と `fe/` の submodule も対になった状態でついてきます。
また、superproject 自体が git リポジトリなので構成をコードとして管理・共有できます。

### git submodule superproject という選択

git submodule の superproject（親リポ）は**アプリのコードを持たない**、純粋に子リポを束ねるためだけのリポジトリです。

- `myapp-be` は今まで通り独自の PR・CI・デプロイを持ちます
- `myapp-fe` も同様に完全独立です
- `myapp`（superproject）は2つの submodule を参照するポインタのみを持ちます

子リポのコードを変更するときは引き続き各子リポの中で PR を出します。superproject は**AI エージェントが両リポを一度に参照するためのウィンドウ**として機能します。

## superproject の構造

### ディレクトリ構成

```
myapp/          ← superproject（アプリコードなし）
├── be/               ← submodule: myapp-be (main ブランチ追跡)
├── fe/            ← submodule: myapp-fe (main ブランチ追跡)
├── .gitmodules
├── CLAUDE.md         ← AI agent 向け横断ルール
└── docs/
```

### .gitmodules の設定

```ini
[submodule "be"]
    path = be
    url = ../myapp-be.git
    branch = main

[submodule "fe"]
    path = fe
    url = ../myapp-fe.git
    branch = main
```

`branch` は `git submodule update --remote` 実行時に参照するブランチを指定します。通常の `--init` では記録済みのコミットに checkout するため、この設定だけで常に最新を追うわけではありません。

**URL を相対パス（`../myapp-be.git`）にしているのがポイントです。**
ローカルで SSH clone していれば SSH、CI で HTTPS checkout していれば HTTPS と、
親リポのプロトコルに自動追従します。絶対 URL にすると CI のみ壊れる、といった事態を防げます。

### clone 時のコマンド

superproject を clone してから submodule を初期化する場合：

```sh
git clone git@github.com:your-org/myapp.git
cd myapp
git submodule update --init --recursive
```

または最初から submodule ごと clone する場合：

```sh
git clone --recurse-submodules git@github.com:your-org/myapp.git
```

## ポインタ自動更新

submodule superproject の最大の運用課題は「submodule のポインタが古くなること」です。
子リポで開発が進んでも、superproject が参照するコミットは手動で更新しない限り古いままになります。

この問題を GitHub App + GitHub Actions で完全に自動化しました。

### 全体の流れ

```
[myapp-be: main へ push]
        │
        │  notify-superproject.yml
        │  App token で repository_dispatch を送信
        │
        ▼
[myapp: update-submodule.yml]
  1. GitHub App token を発行（3リポへのアクセス権）
  2. submodules 込みで checkout
  3. git submodule update --remote -- be/
  4. 差分があれば PR 作成 → auto-merge 有効化
```

`myapp-be` の `main` ブランチへの push が起点となり、
superproject の submodule ポインタ更新 PR が自動作成・auto-merge されるまでが1つの連鎖です。

### なぜ GITHUB_TOKEN ではなく GitHub App か

GitHub Actions のデフォルトトークン（`GITHUB_TOKEN`）には2つの制約があります。

**1. クロスリポジトリアクセスができない**

`myapp-be` のワークフローから `myapp`（superproject）へ `repository_dispatch` を送信したり、
superproject のワークフローから private な `myapp-be` を checkout したりするには、
別リポへのアクセス権が必要です。`GITHUB_TOKEN` は自リポにしかアクセスできません。

**2. GITHUB_TOKEN が作成した PR は CI が起動しない**

GitHub は無限ループ防止のため、`GITHUB_TOKEN` による push 由来のワークフローは起動しません。
PR の CI は承認待ち状態になるため、auto-merge に必要な必須チェックが永久に pending になってしまいます。

**GitHub App の installation token はこれらをどちらも解決します。**
インストール先のリポジトリ（`myapp`・`myapp-be`・`myapp-fe`）へのスコープ付きトークンを発行でき、
App が作成した PR には CI が通常どおり起動します。

### 送信側：notify-superproject.yml（be / fe それぞれに配置）

```yaml
name: Notify superproject (submodule pointer)

on:
  push:
    branches:
      - main

jobs:
  notify:
    runs-on: ubuntu-latest
    steps:
      - name: Generate app token (myapp)
        id: app-token
        uses: actions/create-github-app-token@v3
        with:
          app-id: ${{ secrets.APP_ID }}
          private-key: ${{ secrets.APP_PRIVATE_KEY }}
          owner: ${{ github.repository_owner }}
          repositories: myapp # superproject への dispatch 権限のみ

      - name: Dispatch submodule-update to superproject
        env:
          GH_TOKEN: ${{ steps.app-token.outputs.token }}
          OWNER: ${{ github.repository_owner }}
        run: |
          printf '%s' '{"event_type":"submodule-update","client_payload":{"repo":"myapp-be"}}' > payload.json
          gh api -X POST "/repos/${OWNER}/myapp/dispatches" --input payload.json
```

このワークフローは `myapp-be`・`myapp-fe`（どちらも `main` 追跡）の
それぞれに配置します。どちらもデフォルトブランチへの push をトリガーにして superproject に通知を送ります。

### 受信側：update-submodule.yml（superproject に配置）

```yaml
name: Update submodule pointer

on:
  repository_dispatch:
    types: [submodule-update]
  workflow_dispatch: # 手動起動も可能
    inputs:
      repo:
        type: choice
        options: [myapp-be, myapp-fe]

concurrency:
  group: submodule-update # 固定値にすることで be・fe を直列化して競合を防ぐ
  cancel-in-progress: false

jobs:
  update:
    runs-on: ubuntu-latest
    steps:
      # 1. App token 発行（3リポへのフルアクセス）
      - uses: actions/create-github-app-token@v3
        id: app-token
        with:
          app-id: ${{ secrets.APP_ID }}
          private-key: ${{ secrets.APP_PRIVATE_KEY }}
          repositories: |
            myapp
            myapp-be
            myapp-fe

      # 2. submodules 込みで checkout
      - uses: actions/checkout@v4
        with:
          submodules: recursive
          token: ${{ steps.app-token.outputs.token }}

      # 3. 追跡ブランチの最新へポインタを進める
      - run: git submodule update --remote -- "$TARGET_PATH"

      # 4. 差分があれば PR 作成
      - uses: peter-evans/create-pull-request@v6
        with:
          token: ${{ steps.app-token.outputs.token }}
          branch: submodule-update/${{ steps.target.outputs.path }}
          commit-message: "chore: bump ${{ steps.target.outputs.path }} to ${{ steps.update.outputs.new_sha }}"

      # 5. auto-merge 有効化
      - run: gh pr merge "$PR" --squash --auto
```

`concurrency` の `cancel-in-progress: false` がポイントで、
be と fe が同時に push された場合にポインタ更新が競合しないよう直列化しています。

## AI agent から見た世界

ここまでで superproject のポインタが常に最新に保たれる仕組みが整いました。実際に AI agent からはどう見えるかを確認します。

`myapp` superproject をルートとして Claude Code を開くと、
`be/` と `fe/` が同じディレクトリツリーに見えます。
エージェントはバックエンド実装とフロントエンドを**同時に参照**できる状態になります。

superproject のルートに置いた `CLAUDE.md` には横断開発のルールを記述しています。
「コードの変更・コミット・PR は各 submodule の中で行う」
「両リポをまたぐ機能は be → fe の順で実装する（fe は be の API に依存するため）」
といったルールを書いておくことで、エージェントが superproject のソースを直接編集するような
ミスを防ぎつつ、横断的な調査・開発を一貫して行えるようになります。

## まとめ

git submodule superproject を使うことで、既存の CI/CD・PR・ブランチ運用を一切変えずに AI エージェントの横断開発環境を実現できました。
superproject をルートとして開くだけで be・fe を横断した調査・実装が可能になります。
注意点は2つあります。1つ目は submodule の扱いへの慣れで、`git submodule update --init` を忘れると古いコードを参照してしまいます。2つ目はコードの変更を superproject ではなく各 submodule の中で行う規律です。
複数リポ構成を維持しながら今すぐ AI エージェントの横断開発を始めたいチームに向いている構成です。
